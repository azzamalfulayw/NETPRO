🔐 Keycloak Migration Technical Record

A detailed engineering log documenting the shift from local ASP.NET Identity to external OIDC.

1. Migration Overview

This document records the systematic transition of the NETPRO API from a legacy monolithic authentication model to a modern, decentralized identity architecture. The primary objective was to offload credential management, token issuance, and multi-factor authentication (MFA) capabilities to Keycloak, transforming the .NET backend into a specialized Resource Server.

2. Why Keycloak Replaced Local Authentication

The previous architecture relied on Microsoft.AspNetCore.Identity and a custom TokenService. This setup presented several engineering challenges:

  • Security Overhead: The API was responsible for password hashing, salt management, and credential storage, increasing the attack surface.
  • Scalability Limitations: Local identity tables coupled user data with authentication logic, making it difficult to implement SSO across multiple future services.
  • Feature Gap: Implementing advanced features like social login, user sessions, or automated email verification required significant manual coding.

Keycloak was adopted to provide an industry-standard OpenID Connect (OIDC) implementation, offering enterprise-grade security and a centralized identity management console.

3. Architecture Before and After

🔴 Before (Legacy)

The API acted as the Identity Provider and the Resource Server.

  • Database: 7+ Identity tables (AspNetUsers, AspNetRoles, etc.).
  • Service: TokenService.cs manually generated JWTs.
  • Login: API handled username/password directly JSON-over-HTTP.
  • Validation: Validated tokens using a locally stored symmetric Key.

🟢 After (Modern)

The API acts strictly as a Resource Server.

  • Authority: Keycloak manages all credentials and sessions.
  • Database: Zero Identity tables. AppUser is a simple domain mapping.
  • Login: OAuth2 Authorization Code flow (browser-based redirect).
  • Validation: Validates tokens using Keycloak's public keys via discovery (`.well-known`).

4. Authentication and Request Flow

The new flow follows the Authorization Code Flow with PKCE for the Swagger UI development environment.

  1. Swagger Login: User clicks "Authorize," redirecting them to the Keycloak login page (localhost:8080).
  2. Token Issuance: Upon successful login, Keycloak issues an Access Token (JWT).
  3. API Request: The browser sends the JWT in the Authorization: Bearer [token] header.
  4. JWT Validation: The .NET middleware validates the signature, issuer, and audience from the token.
  5. User Resolution: The UserResolverService extracts the sub claim. If the user doesn't exist locally, it auto-provisions an AppUser row to maintain database relationships.

5. Implementation Details by File

The following files were central to the technical transition:

📄 Program.cs

Modified Replaced AddIdentity with AddAuthentication and AddJwtBearer. Configured Swagger to support OAuth2 security definitions.

// Technical Configuration for JWT Validation
builder.Services.AddAuthentication(options => {
    options.DefaultAuthenticateScheme = JwtBearerDefaults.AuthenticationScheme;
})
.AddJwtBearer(options => {
    options.Authority = "http://localhost:8080/realms/netpro";
    options.Audience = "account,netpro-api,netpro-swagger";
    options.TokenValidationParameters = new TokenValidationParameters {
        ValidateIssuer = true,
        ValidIssuer = "http://localhost:8080/realms/netpro",
        NameClaimType = "preferred_username",
        RoleClaimType = "roles"
    };
});

📄 UserResolverService.cs

Added Acts as the "bridge" between the external Keycloak token and local database entities.

public async Task<AppUser?> GetUserAsync() {
    var sub = user.FindFirst(ClaimTypes.NameIdentifier)?.Value;
    var appUser = await _context.AppUsers.FirstOrDefaultAsync(u => u.Id == sub);

    if (appUser == null) {
        appUser = new AppUser { Id = sub, UserName = "..." }; // Auto-provisioning logic
        _context.AppUsers.Add(appUser);
        await _context.SaveChangesAsync();
    }
    return appUser;
}

Modified CommentController.cs and RatingController.cs were updated to use [Authorize] and ownership logic.

public async Task<IActionResult> Delete(int id) {
    var appUser = await _userResolverService.GetUserAsync();
    var existingResource = await _mediator.Send(new GetByIdQuery { Id = id });

    if (existingResource.AppUserId != appUser.Id) return Forbid();
    // ... delete logic
}

6. Legacy Components Removed

The following technical debt was fully eliminated from the repository:

  • AccountController / AccountContriller: Removed entire logic for /login and /register. These endpoints are now unnecessary as Keycloak provides the UI for these flows.
  • TokenService / ITokenService: Removed the manual JWT generation logic. The API no longer possesses private keys for token signing.
  • Microsoft.AspNetCore.Identity Packages: Removed Identity.EntityFrameworkCore and Identity.Core. This prevents the "Identity Schema" tables from being generated in migrations.

7. Authorization and Ownership Hardening

We implemented a strict security matrix to ensure data privacy.

Endpoint Public? Ownership Check? Reasoning
GET Stock/Comments/Ratings ✅ Yes ❌ No Read-only community feed should be open.
POST (Creation) ❌ No ❌ No Anyone authenticated can contribute.
PUT (Update) ❌ No ✅ Yes Prevents users from editing others' comments.
DELETE ❌ No ✅ Yes Only the original author can remove data.

8. Issues Encountered During Migration

⚠️ Compile Error: Missing [Authorize] Namespace
After applying authorization attributes to CommentController.cs, the build failed because Microsoft.AspNetCore.Authorization was not imported. This was a direct result of the cleanup that removed local Identity dependencies. FIXED by adding the explicit import.
ℹ️ Challenge: Multi-Audience Workaround
Keycloak tokens by default often include account in the audience, while our API expected netpro-api. This caused 401 Unauthorized errors in early testing. WORKAROUND: Modified appsettings.json to allow a CSV-style list of valid audiences (account,netpro-api,netpro-swagger).

10. Remaining Risks and Recommendations

While the migration is technically sound for development, the following items are recommended for the next sprint:

  • Strict Audience Validation: Transition from the multi-audience workaround to a proper Keycloak "Audience Mapper" to keep tokens specific to the API.
  • Role Management (RBAC): Implement a claim transformation to map Keycloak "Roles" to .NET Claims so that `[Authorize(Roles = "Admin")]` can be utilized.
  • Nullable Cleanup: Address naming warnings like Task<Rating> in RatingRepository.cs to align with modern C# nullable standards.
