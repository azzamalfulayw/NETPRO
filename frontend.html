🔐 Authentication Migration (Manual to OIDC)

Why Manual Token Input was Removed

Phase 6 Entry

The previous approach required users to manually paste JWT strings into a textarea. This was deprecated for three reasons:

  • Security: Storing raw JWTs in plain text in localStorage without a managed flow increases the risk of XSS payload theft.
  • User Experience: Tokens expire frequently. Manually re-fetching tokens from Swagger/Keycloak every 60 minutes prevented a smooth continuous workflow.
  • Protocol Integrity: OAuth2 defines standard redirect flows for a reason. Replacing manual input ensures the app behaves like a modern SPA.

Technical Flow: Auth Code + PKCE

We chose the Authorization Code Flow with PKCE (Proof Key for Code Exchange) as it is the current security standard for Single Page Applications.

  • How Login Works: User is redirected to Keycloak. After login, Keycloak redirects back with a code. The app exchanges this code for a token natively.
  • How Logout Works: authService.logout() clears the local React state and triggers a redirect to the OIDC end-session endpoint, returning the user to the Login page.
  • Route Protection: A ProtectedRoute.tsx Higher Order Component (HOC) wraps all feature routes. It checks the isAuthenticated status and redirects to /login if the session is absent.
  • Token Refresh: Axios interceptor calls authService.updateToken(30) before every request. If the token has less than 30 seconds left, it is refreshed silently via silent-check-sso.html.

📈 Verified Behavior & Fail-Safes

  • ✅ Unauthenticated Access: Visiting /dashboard directly when logged out correctly triggers a redirect to /login.
  • ✅ Successful Callback: Post-Keycloak login, the browser returns to the app and automatically hydrates the AuthContext.
  • ✅ Bearer Injection: Network inspection confirms Authorization: Bearer is present on all stock/transaction calls.
  • ✅ Expired Sessions: If the refresh token expires or is revoked, the app now redirects the user back to the login page instead of displaying empty charts.
  • ⚠️ Keycloak Offline: If the Keycloak server is down, the app displays a "Initialization Failed" error in the console and blocks access to ensure data safety.
  • ⚠️ API 401/403: If the API rejects a valid-looking token, the app maintains the current page but displays a toast notification (react-hot-toast).

🛠️ Debugging & Fixes

Real-time log of frontend troubleshooting and parity fixes.

Fix 1: API Port Mismatch (5246 → 5247)

2026-04-02

Problem: All frontend features showed "Connection Error" despite the backend being online.

Root Cause: The launchSettings.json in the backend was configured to use port 5247 for HTTP, but the frontend was defaulting to 5246.

Fix: Updated api.ts base URL to http://localhost:5247/api.

Result: Connectivity restored. Network requests now reach the backend controllers.

Fix 2: JSON Casing Incompatibility

2026-04-02

Problem: Data fetched successfully but UI displayed "Failed to aggregate analytics" or empty values.

Root Cause: The .NET backend uses Newtonsoft.Json with default PascalCase settings (returning Symbol, CurrentPrice), while the React frontend expects camelCase (symbol, currentPrice).

Fix: Implemented a Recursive Response Interceptor in api.ts to automatically convert all incoming JSON keys to camelCase before they reach the React components.

Result: All dashboard charts and stock cards are now correctly hydrated with real data.

Fix 3: Enhanced Error Telemetry

2026-04-02

Implemented: Improved Axios error interceptors to provide precise logging for 401 Unauthorized, 403 Forbidden, and Network Failures.

Why: To allow for faster troubleshooting of Keycloak audience mismatches or backend offline states.

Fix 4: Comprehensive Trading Workflow

2026-04-02

Problem: Transactions page was read-only; executions failed due to Symbol/StockId mismatch.

Root Cause: The backend CreateTransactionDto requires an integer StockId, but the UI only collects the Symbol string. Also, transaction Categories and Types were misaligned with backend enums.

Fix: Developed a Two-Step Execution Pipeline in the frontend:

  1. Id Resolution: The frontend first queries the /stock endpoint to resolve the symbol into its internal id.
  2. DTO Mapping: Maps the resolved ID and UI state into the exact CreateTransactionDto (including integer enums for Market/Limit orders).
  3. Synchronized Invalidation: Automatically triggers a cache refresh for Transactions, PortfolioPerformance, and RealizedGains upon success.

Result: Users can now buy and sell stocks directly from the Ledger. The dashboard balance and gains cards reflect these trades immediately.

Fix 5: API Port Unification & Migration

2026-04-02

Problem: Desync between SettingsPage (5246) and api.ts (5247) caused silent connectivity failures.

Root Cause: Legacy hardcoded defaults in the Settings feature.

Fix: Unified all defaults to 5247. Added an automatic migration interceptor in api.ts that detects and updates localStorage if it contains the old 5246 value.

Fix 6: Dynamic API Connectivity Status

🏗️ Integrated Trading & Execution Flow

The "Invisible Logic" Problem

2026-04-05

Root Cause: Previously, "Buy" and "Sell" actions were secondary. Users had to navigate to the Transactions Ledger and manually type symbols, quantities, and prices. This created a high friction environment where the most critical user action (trading) was buried under a "Log Execution" button.

Design Decision: We refactored the trade flow into a Single Source of Truth. Instead of duplicating mutation logic, we created a centralized hook and a polymorphic modal that adapts to the entry point (Marketplace, Watchlist, Portfolio, or Ledger).

Wording Evolution: Replaced technical jargon like "Log Execution" with intent-based wording: Buy, Sell, and Confirm Order.
